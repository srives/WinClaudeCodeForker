### Phase 19: Dead Session Detection & CLI Compatibility (v2.1.1 - February 24, 2026)
**Scope: Session Lifecycle & Resilience**

Claude Code silently deletes old session `.jsonl` files over time, leaving orphaned index entries. This phase adds detection and cleanup for these "dead" sessions, plus fixes a breaking change in the Claude CLI binary.

- **Skull marker (☠)**: Dead sessions (whose `.jsonl` files no longer exist on disk) now show a skull in the Active column, giving immediate visual feedback that Claude has garbage-collected the conversation
- **Purge dead sessions (P key)**: Bulk archive or bulk delete all dead sessions in one operation, keeping the session list clean
- **Claude .exe compatibility fix**: On Feb 23, 2026, Claude CLI updated from a `.cmd` Node.js shim to a native `claude.exe` binary. Windows Terminal cannot pass .exe paths with arguments through its command line. New `Start-WTClaude` function writes the full command into the WT profile's `commandline` field in settings.json before launching
- **Column sort fix**: Sort keys now map to the Nth VISIBLE column instead of hardcoded column numbers. `$Global:ColumnDefinitions` master array is the single source of truth for all column metadata (header, width, sort property). Same column press toggles direction; new column starts ascending
- Bug fixes: Enter key no longer confirms bulk deletion in Purge (only Y), unique `.cmd` launch filenames prevent race conditions, WT settings cache invalidated after writes

### Phase 21: Quality & Resilience Overhaul (v3.1.0 - February 26, 2026)
**Scope: Critical Bug Fixes, Platform Abstraction, Comprehensive Test Expansion, Cross-Platform Fixes**

This phase represents a major quality investment — a systematic code audit that uncovered 2 critical production bugs, drove platform abstraction improvements, and expanded the test suite by 58% (158→250 tests). The result is significantly more resilient software with test coverage reaching into areas the original 158 tests never touched.

**Critical Bug Fixes:**
- **WT Backup Restore Fix (CRITICAL):** `Test-Json -Path $backupPath` in both `Add-WTProfile` and `Remove-WTProfile` catch blocks was broken on ALL PowerShell versions. PS 5.1 has no `Test-Json` cmdlet (throws), PS 7+ has no `-Path` parameter (returns false). If WT profile add/remove failed mid-write, the backup existed but restore always silently failed, leaving WT settings corrupted. Replaced with `ConvertFrom-Json` validation that works on all PowerShell versions.
- **Codex Prefix Strip Fix:** 10 hardcoded `$WTProfileName -replace '^Claude-', ''` locations throughout the codebase silently failed for Codex profiles (`Codex-MySession` → `Codex-MySession` unchanged). This caused background images to go to wrong directories, deletions to miss files, and diagnostics to show wrong data. Added `Get-SessionNameFromWTProfile` utility function and replaced all 10 locations.

**New Utility Function:**
- `Get-SessionNameFromWTProfile` — iterates `Get-AllWTProfilePrefixes()` to strip the correct platform prefix from any WT profile name, future-proofing for additional CLI integrations

**Codex New Session WT Integration:**
- Codex new sessions now receive full Windows Terminal integration: named Codex sessions get background image generation, `Codex-` prefixed WT profiles, and launch with profile. Previously the Codex new-session path had an early return that bypassed all WT integration with the comment "skip model choice, WT profiles, etc."
- Unnamed Codex sessions still launch with default profile for quick start

**Test Suite Expansion (158 → 250 tests, +58%):**

## New Features: AI CLI Integration Candidates

*Analysis date: February 25, 2026*

This section evaluates other AI coding CLI tools for potential integration into SessionForge, the same way we integrated Claude Code and Codex. For each tool we assess:

- **Compatibility Score (0-100):** How well the tool's session storage maps to our existing architecture (session discovery → unified menu → continue/fork dispatch). Higher = easier to integrate.
- **Value Score (0-100%):** How much incremental product value the integration adds (user base expansion, feature differentiation, market positioning).
- **Confidence Level:** Whether we have verified the file structures firsthand or are relying on documentation that may be incomplete or outdated.

### Why Claude Code + Codex Worked So Well Together

Both Claude Code and Codex share a critical pattern that made integration straightforward:

1. **Discoverable session storage** — Claude uses `.jsonl` files indexed by `sessions-index.json`; Codex uses a SQLite `threads` table. Both are readable without running the CLI.
2. **Resume by ID** — `claude --resume <UUID>` and `codex resume <UUID>`. Identical pattern.
3. **Fork by ID** — `claude --resume <id> --fork-session` and `codex fork <id>`. Both support branching.
4. **Session metadata available offline** — title, timestamps, model, tokens, project path — all stored in the session data, not requiring an API call to retrieve.
5. **Stateless CLI launch** — both CLIs can be launched with arguments from Windows Terminal without needing a running daemon or server.

Not all tools share these properties. The key question for each candidate: **Can we discover sessions, read metadata, and launch resume/fork without the CLI running?**

| Metric | Assessment |
|--------|------------|
| **Compatibility** | 25/100 |
| **Value** | 40% |

**Session Storage:** Markdown file (`.aider.chat.history.md`) in project directory. Input history in `.aider.input.history`. No session IDs, no database, no index file.

**Resume:** `--restore-chat-history` reloads the last conversation. No resume-by-ID. No session listing command.

**Fork:** None. Aider's philosophy is git-centric — you branch code, not conversations.

**Integration Challenges:**

- No session index — we'd have to scan every project directory for `.aider.chat.history.md` files

- Resume is "reload last chat," not "resume session X" — we can't dispatch to a specific session

**What Would Work:** We could discover Aider history files, show them in the menu with project path and modification date, and launch `aider --restore-chat-history` in the correct directory. But no fork, no session selection, no cost tracking.

**Verdict:** Poor fit. Aider's "one history per project" model doesn't map to our multi-session architecture. Would feel like a second-class integration.

**MUST VERIFY:** Whether `.aider.chat.history.md` contains parseable metadata (model, tokens) or is purely conversational markdown.
